# Overview

Write website copy that is clear, technically credible, and commercially useful.

The goal is not generic SaaS copy.
The goal is to explain UCE in a way that engineers and technical buyers trust.

## Core rules

- Write for technically literate readers first.
- Keep claims specific and believable.
- Prefer clarity over cleverness.
- Use strong nouns and verbs.
- Avoid hype, fluff, and vague abstractions.
- Keep sentences tight.
- Preserve terminology consistency.

## Preferred language style

Use language such as:
- structured control definition
- deterministic validation
- canonical engineering model
- reviewable outputs
- downstream implementation
- engineering workflow
- controls-definition process
- model-driven automation engineering

Avoid language such as:
- revolutionary
- game-changing
- cutting-edge AI disruption
- seamless synergies
- next-generation innovation platform

## Audience priorities

### EPCs
Care about:
- reviewability
- standardization
- reduced rework
- better engineering consistency
- lower commissioning risk

### Controls engineers
Care about:
- sequence clarity
- interlocks
- validation
- deterministic outputs
- practical workflows

### Process engineers / technical leads
Care about:
- traceability
- alignment between process intent and controls implementation
- reduced ambiguity
- structured collaboration

## Homepage copy pattern

Use this structure unless a stronger one is clearly better:

1. Headline
2. Supporting subheadline
3. Primary CTA
4. Problem statement
5. UCE solution statement
6. Key capabilities
7. Why it matters / business outcomes
8. Technical credibility section
9. Final CTA

## Headline guidance

Strong headlines:
- say what UCE is
- mention the problem category
- avoid empty slogan writing

Good direction:
- Structured engineering for industrial control systems
- Define control intent once. Validate it deterministically.
- A deterministic engineering compiler for industrial automation workflows

Weak direction:
- Reimagining the future of industrial excellence
- The next era of smart automation
- AI-powered transformation for everything

## CTA guidance

Preferred CTA tone:
- direct
- credible
- low-hype

Examples:
- Request early access
- Talk to the founder
- See the platform vision
- Review the architecture
- Join the pilot list

## Feature/capability writing

When writing feature blocks:
- state the capability
- state why it matters
- keep each block short
- avoid repeating the same meaning

Pattern:
- Capability title
- One-sentence explanation
- One practical outcome

## Technical credibility rule

When describing architecture:
- keep the company/platform/product hierarchy accurate
- do not invent capabilities
- do not overpromise roadmap items as current product state
- phrase future items carefully

## Output preference

When drafting copy, provide:
- a recommended version
- optional shorter alternatives for headline/CTA when useful
- concise rationale only when needed
